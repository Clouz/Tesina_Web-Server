Web Server in Java utilizzando i Socket SSL
L'importanza di una comunicazione sicura attraverso internet

Scopo del progetto

Lo scopo principale di questo progetto è applicare quanto imparato, sia sulla cifratura, sia sui Socket SSL, per la creazione di un semplice Web Server HTTPS. Le sue funzionalità principali devono comprendere il leggere e l’interpretare una richiesta HTTP (HTTP Request), ricavandone quanto necessario per generare una risposta HTTP (HTTP Response). Tutto questo attraverso un canale sicuro (SSL/TLS)

< Precedente Prossimo >

Un canale sicuro attraverso internet (SSL/TLS)

Il protocollo SSL/TLS è nato al ﬁne di garantire la sicurezza delle trasmissioni su Internet mediante tre funzionalità fondamentali:

Riservatezza del collegamento

Autenticazione

Affidabilità

La riservatezza del collegamento viene garantita mediante algoritmi di crittograﬁa a chiave simmetrica (ad esempio DES e AES).

L’autenticazione dell’identità viene effettuata con la crittograﬁa a chiave pubblica (per esempio RSA): in questo modo si garantisce ai client di comunicare con il server corretto, introducendo a tale scopo anche meccanismi di certiﬁcazione sia del server che del client.

Il livello di trasporto include un controllo sull’integrità del messaggio con un sistema detto MAC (Message Authentication Code) che utilizza funzioni hash sicure come SHA e MD5: avviene la veriﬁca di integrità sui dati spediti in modo da avere la certezza che non siano stati alterati durante la trasmissione.

Ciclo di vita del web server

La struttura del Web Server è riassumibile in 4 passaggi:

creaServer() > connessione() > clientHeader() > inviaFile()

La creazione del Server Socket (creaServer()) viene eseguita solo all’avvio del Web Server, mentre mettersi in attesa del client (connessione()), leggere la richiesta (clientHeader()) e inviare la risposta (inviaFile()) sono eseguiti in un ciclo senza fine.

< Precedente Prossimo >

Creazione del server socket

In questo primo passo viene creato il Server Socket attraverso la classe SSLServerSocketFactory. Questa classe permette di rendere del tutto trasparente l'Handshake effettuato dal protocollo TLS.
E' necessario specificare anche la posizione delle coppie di chiavi (keystore) e dei certificati digitali (truststore) con le relative password.

Questa è l’unica differenza che si può trovare tra un Socket SSL ed un Socket TCP in chiaro, in quanto sarà SSLSocketFactory ad occuparsi di tutti i dettagli della configurazione di un Secure Socket.

< Precedente Prossimo >

Attesa del client

Il secondo passo consiste nel mettersi in attesa del Client. Questo metodo connessione() rappresenta il corpo del programma e da qui verranno gestite tutte le successive fasi.

Il server è in attesa sulla porta 443 (porta di default per HTTPS), successivamente un client richiede una connessione ed attraverso il protocollo TLS viene instaurata una comunicazione sicura.

< Precedente Prossimo >

HTTP Request

In questo terzo passo viene effettuata la lettura dell'Header HTML, dove la prima riga conterrà URI della risorsa richiesta. Ad esempio attraverso il metodo GET il client fa una richiesta della risorsa “/index.html” utilizzando il protocollo HTTP versione 1.1. Le successive righe della richiesta si chiamano Request Header e forniscono informazioni aggiuntive come ad esempio l’IP dell’host e l’User-Agent.

< Precedente Prossimo >

HTTP Response

A questo punto si può procedere ad inviare la risposta al client, impacchettando il messaggio come richiesto dal protocollo HTTP (HTTP Response message). Esso è composto da una Status Line e da un Response Header. Dopo una riga bianca sarà aggiunto il corpo del messaggio; in questo caso, il file ricavato precedentemente sarà letto dalla classe FileInputStream e scritto sul Buffer di uscita verso il client.

Verificare l'identità

La verifica dell'identità viene effettuata attraverso un certificato digitale contenente la chiave pubblica del Web-Server, se questo cambiasse il browser ci avvertirebbe immediatamente.

Verificare la privacy

Per verificare che il canale di comunicazione garantisca la privacy, si utilizzerà il packet sniffer Wireshark, con lo scopo di visionare da prima il traffico generato da un Web Server HTTP e successivamente con la stessa metodologia si verificherà la versione HTTPS.

Conclusioni

Attualmente il protocollo TLS risulta inviolato, ma questo non significa che lo sarà per sempre, quindi è molto importante rimanere sempre aggiornati sull’evoluzione della crittografia, per non rischiare che i nostri dati finiscano nelle mani sbagliate.
